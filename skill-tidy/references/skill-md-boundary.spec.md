# `SKILL.md` 结构与边界

当你需要判断维护对象的 `SKILL.md` 应保留什么、移出什么时，使用这份 reference。

## `SKILL.md` 的职责

`SKILL.md` 是 skill 的激活时说明文件。agent 判断该 skill 相关后，这个文件应帮助它立即知道：

1. 这个 skill 负责什么。
2. 执行它需要什么输入。
3. 应按什么顺序工作。
4. 接下来该读或该运行哪些附属文件。
5. 如何判断工作已经完成。

若某段内容对以上任一目标都没有帮助，就应重新判断它是否该留在这里。

## 推荐结构

正文没有强制固定模板，但一个便于长期维护的 `SKILL.md` 通常包含：

1. 标题和一句话用途。
2. 范围或目标。
3. 输入或前置条件。
4. 工作流程或执行步骤。
5. 对附属文件的引用，以及在什么条件下打开它们。
6. 完成标准或校验项。

顺序保持简洁，避免章节越来越多但每节信息密度很低。

## Frontmatter 放什么

frontmatter 只保留元数据：

- `name`
- `description`
- specification 定义的、确有实际意义的可选字段

不要把流程说明、示例或冗长兼容性说明塞进 frontmatter。

## `SKILL.md` 放什么

主文件中应保留：

- 能完整说明如何应用该 skill 的最短必要指引
- 会影响附属文件选择的决策点
- 每次执行几乎都需要看到的短 checklist
- 会改变执行行为的关键注意事项

## `references/` 放什么

遇到以下内容时，应优先移入 `references/`：

- 细节较重，每次都加载会浪费上下文
- 具有规范性，但只会在某些维护步骤中查阅
- 更长的边界条件、规则目录或格式要求
- 有助于理解但不是每次执行都必须阅读的示例

每个 reference 文件都应由 `SKILL.md` 显式链接。

## `scripts/` 放什么

当逻辑具有以下特征时，应移入 `scripts/`：

- 追求确定性
- 会反复复用
- 运行脚本比反复用文字重述更合适

`SKILL.md` 应说明何时运行这些脚本，以及预期结果是什么。

## `assets/` 放什么

当文件是静态资源，且其用途是被复制、嵌入或作为输出的一部分被消费，而不是被当作说明阅读时，应移入 `assets/`。

## 通常应删除的内容

以下内容很容易造成长期漂移：

- agent 人设或对话风格要求
- 与 skill 真实任务无关的产品 UI 指引
- 本应放在 skill 之外的仓库级开发规范
- 变更记录式历史信息
- 关于“这个 skill 是如何被写出来的”审校笔记
- 已经在某个 reference 中完整表达的重复说明

## 边界判断问题

审查 `SKILL.md` 中某一段时，依次问：

1. agent 在激活后是否立刻需要看到它？
2. 它是否会改变“下一步该看哪个文件”或“当前该执行什么步骤”？
3. 把它移到 `references/` 后，主文档是否会更清晰且不影响执行？

如果前两个问题的答案都是否，且第三个问题的答案是是，就应把这段内容移出 `SKILL.md`。
