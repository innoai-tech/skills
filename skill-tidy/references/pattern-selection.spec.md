# Skill 模式判定规范

用于判断一个 skill 的主设计模式。

## 目标

1. 为目标 skill 判定一个主模式。
2. 优先把目标 skill 收敛为单一主模式。
3. 避免把多个模式的主流程混写到同一个 `SKILL.md`。
4. 让附属资源的职责服从主模式，而不是为了“看起来完整”堆结构。

## 判定顺序

维护一个 skill 时，依次回答：

1. 它的主要职责是什么？
2. 它依赖什么方式才能稳定执行？

然后：

1. 判定一个主模式。
2. 先检查是否可以只保留这一个主模式。
3. 只有在删除后会损伤执行稳定性时，才记录一到两个辅模式。
4. 用主模式决定 `SKILL.md` 骨架。
5. 用辅模式补局部控制逻辑，不重写全局流程。

若一个 skill 需要两套以上平行主流程，或长期依赖多个辅模式共同维持可读性，通常应拆 skill，而不是继续叠加。

## 可选主模式

1. `Tool Wrapper`
2. `Generator`
3. `Reviewer`
4. `Inversion`
5. `Pipeline`

## 判定规则

### `Tool Wrapper`

当 skill 的主要价值是封装某个库、框架、平台、协议或工具约定时，判为 `Tool Wrapper`。

### `Generator`

当 skill 的主要价值是生成结构稳定、输入驱动的输出时，判为 `Generator`。

### `Reviewer`

当 skill 的主要价值是审查、评估、列问题或给出结论时，判为 `Reviewer`。

### `Inversion`

当错误默认假设会导致高成本返工，且 skill 的关键价值是先问清楚再执行时，判为 `Inversion`。

### `Pipeline`

当 skill 的主要价值依赖顺序步骤、阶段门禁或确认点时，判为 `Pipeline`。

## 混合模式规则

1. 单一主模式是默认目标，混合模式是例外。
2. 只保留一个主模式决定主文档骨架。
3. 辅模式只补必要章节，不改写主流程。
4. 若混合后仍无法保持清晰，应拆成多个 skill。
