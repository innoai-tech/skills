# 项目分型与分流规范

用于按协作结构而不是按编程语言，判断仓库或目录应采用什么控制面组织方式。

## 分型原则

1. 优先按协作边界分型，而不是按语言、框架或目录名分型。
2. 只有独立协作边界，才需要独立控制面。
3. 目录存在本身，不是新增模块级控制面的充分理由。

## root 控制面

适用场景：

- 仓库根目录
- workspace 或 monorepo 的全局层
- 需要统一约束多个模块或多个运行角色

要求：

1. root `AGENTS.md` 承载全局协作约束与基本工程约束。
2. root `README.md` 承载最小概述与文档索引。
3. root `justfile` 聚合已确认的稳定入口与工具链入口。

## workspace

适用场景：

- 某目录本身是一个独立协作边界，但仍处于更大仓库之内

要求：

1. 先确认该目录是否真的构成独立 workspace，而不是普通实现目录。
2. 若成立，可拥有自己的局部控制面。
3. 若不成立，应继续继承上层控制面，而不是补一套新的三件套。

## 模块分型

模块只按协作角色分为两类：

1. `libs`
2. `projects`

### `libs`

适用场景：

- 目录主要提供通用能力、库接口或可复用组件。

要求：

1. README 优先说明能力边界、依赖关系和文档索引。
2. 若已有完善语言级文档，例如 Go doc，README 只保留概述与索引。
3. 执行入口如确有需要，统一收敛到 `justfile`。

### `projects`

适用场景：

- 目录主要表达一个可运行、可交付或可操作的项目单元。

要求：

1. README 优先说明职责、边界和文档索引。
2. 执行入口统一收敛到 `justfile`。
3. `projects` 再分为单入口与多入口。

## 执行面分型

### 单入口

适用场景：

- 当前目录只服务一个稳定目标。

要求：

1. 执行入口集中在单个 `justfile`。
2. 不为同一目标重复铺设多套命令入口。

### 多入口

适用场景：

- 当前目录下有多个同构目标，需要按参数选择入口。

要求：

1. 多个目标通过同一 `justfile` 参数化选择。
2. 不按语言或框架拆成多套平行入口。

### toolchain

适用场景：

- 当前目录的主要职责是聚合某一类语言或构建工具链入口。

要求：

1. 所有语言工具链统一收敛到 `justfile` 暴露。
2. 默认使用 `[no-cd]`。
3. 模块若只是收窄范围，优先复用 toolchain 入口。

## 需要确认的事项

以下事项必须先确认，再落地：

1. 是否新增子模块。
2. 新增子模块采用单入口还是多入口。
3. root `justfile` 是否需要注册新的稳定入口。
4. 某个目录是否应视为独立 workspace、模块或执行面。
