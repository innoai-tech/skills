# README Guideline

用于定义 `README.md` 应如何作为最小概述与索引层存在。

## 目标

1. 让人和 agent 快速知道当前目录负责什么。
2. 提供继续阅读所需的最小索引。
3. 让 root README 作为仓库根入口文档时，仍然清晰、克制、可扫描。
4. 不把 `README.md` 膨胀成规则、命令或设计细节堆积点。

## 最小结构

`README.md` 通常只需要：

1. 标题与一句话概述，先让读者知道这是什么。
2. 一小段背景或价值说明，回答它解决什么问题、为什么存在。
3. 职责与边界，说明当前目录或项目负责什么、不负责什么。
4. 指向源码、文档、示例、上下游或子目录的最小索引。

若内容较多，优先遵循常见 README 阅读顺序：`是什么` -> `为什么存在` -> `怎么开始` -> `去哪里看更多`。不要一上来就铺目录清单。

## root

root `README.md` 首先是仓库根入口文档，而不是内部控制面文件清单。对 root README，优先遵循以下组织方式：

1. 开头先回答仓库是什么、解决什么问题、为什么存在。
2. 若仓库存在明确主入口，优先给出一个最小起步入口，例如主文档、主模块或最常用命令入口。
3. 用经过筛选的导航替代目录枚举，优先列“最值得先看什么”，而不是把所有文件平铺出来。
4. 若仓库是 monorepo 或多 skill / 多模块仓库，优先按职责分组导航，而不是按文件名罗列。
5. 只保留仓库级最小信息密度高的内容；详细设计、协作规则、命令细节和实现原理继续下沉到对应文件。

## 导航写法

1. 索引应优先按“入口 / 主要模块 / 参考资料 / 协作文件”分组，而不是单纯按路径出现顺序堆叠。
2. 每个链接只补一句用途说明，说明“为什么要点进去”，不要重复文件名本身。
3. 对 root README，优先链接稳定入口，例如主文档、子项目 README、核心 guideline、统一执行入口。
4. 只有当读者确实需要直接抵达某个文件时，才链接控制面文件，例如 `AGENTS.md`、`justfile`。
5. 若仓库已经有更合适的主页或完整文档站，README 应显式把读者导向那里，而不是在 root 重复写一遍。

## 表达偏好

1. 优先短段落加少量高价值列表，避免连续的大段目录清单。
2. 优先写“仓库价值与主要入口”，再写“职责与边界”，最后再给扩展索引。
3. 对 root README，允许保留少量能帮助快速判断仓库价值的介绍性内容，不必机械压缩到只剩索引。
4. 若仓库主要面向贡献者或维护者，README 也应先帮助新读者建立最小上下文，再进入维护导航。

## `libs`

1. 优先说明能力边界、依赖关系和文档索引。
2. 若已有完善语言级文档，README 只保留概述与索引。

## `projects`

1. 优先说明职责、边界和相关文档索引。
2. 如需说明运行方式，只保留最小入口，并把具体执行留给 `justfile`。

## 不应包含

- 协作规则
- 规则优先级
- 大量命令细节
- 详细架构推导
- 长期维护的设计细节正文
- 为了“显得完整”而堆砌的原始文件清单
