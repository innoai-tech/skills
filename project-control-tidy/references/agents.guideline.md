# AGENTS Guideline

用于定义 `AGENTS.md` 在满足强约束之外，通常应如何组织，才能更稳定地约束 agent 决策。

## 目标

1. 让 agent 快速判断哪些约束对当前路径生效。
2. 让多层 `AGENTS.md` 的继承与覆盖关系清晰可裁决。
3. 让约束尽量短、硬、稳定，不被说明性内容稀释。

## 建议结构

`AGENTS.md` 通常只需要：

1. 作用域。
2. 规则优先级。
3. 变更边界。
4. 停止条件。
5. 人工接管条件。
6. 文件边界。
7. 基本工程约束。
8. 规则启用机制。

## 规则启用机制

1. `AGENTS.md` 应明确只有落盘到 `AGENTS.md` 的规则才生效。
2. 临时口头要求、对话中的即兴偏好或未落盘说明，不应视为仓库协作规则。
3. 若仓库采用多层 `AGENTS.md`，建议在 root 显式写出启用机制；下层若未重写，则继续继承。

## root

1. root `AGENTS.md` 负责定义全局协作约束。
2. root 只保留跨任务都成立的基本工程约束，例如任务拆分、架构分层、测试分层习惯。
3. root 不应展开具体 workflow、命令细节或项目背景。
4. root 应显式写出规则启用机制。

## coding projects

对编码类项目，`AGENTS.md` 应覆盖一组可直接裁决 agent 行为的基线约束。基线约束的核心语义必须保持一致，可做等义压缩，但不应做语义漂移式改写；项目特化要求可另补，但不应替换或削弱基线。

### 起手动作

1. 先做最小盘点，再开始改动。

### 变更范围

1. 只修改与当前任务直接相关的仓库内文件。
2. 不做与任务无关的功能扩展、顺手重构或批量风格清理。
3. 不因目录已存在而默认新增模块、执行入口或注册关系。
4. 不执行破坏性历史改写，不回滚或覆盖自己未引入的变更，除非得到明确确认。

### 验证与说明

1. 能完成最小相关验证时，必须先验证再交付。
2. 无法完成最小验证时，必须明确说明缺口、原因与风险。
3. 测试描述、注释、错误或日志说明应尽量直接、准确、可验证。

### 架构与实现

1. 优先通过参数或依赖注入传递上下文，避免可变全局状态。
2. 保持基本架构分层，不为局部便利破坏长期一致性。
3. 避免过度工程、过度防御、过早抽象与无必要注释。

### 协作立场

1. 保持严谨、专业、客观，以代码正确性、架构一致性和长期可维护性为先。
2. 尽量使用中文，除非专用名称，包括但不限于文档、注释、错误和日志信息。
3. 当用户要求与代码正确性、架构约束或维护成本明显冲突时，必须先指出风险，再讨论可接受折中，而不是直接顺从。
4. 交付说明只陈述已完成并已验证的事实；未验证项、假设项与风险项必须显式区分。

### 暂停与确认

1. 当需求边界不清且存在多种互斥实现时，必须先暂停并请求确认。
2. 当需要新增模块、入口、注册关系、workspace 边界或执行面模式时，必须先暂停并请求确认。

### 结构异常

1. 若发现项目结构、模块分层、入口组织或控制面职责不符合预期，必须明确说明，并给出可执行的重构建议。

## nested

1. 子目录 `AGENTS.md` 只重写确实需要局部化的约束。
2. 未被下级重写的约束继续继承上级。
3. 冲突约束由更近的 `AGENTS.md` 覆盖。

## 表达要求

1. 每条规则都应直接影响 agent 决策。
2. 优先使用可裁决语言，不写口号式表述。
4. 不把 README 语义、命令说明或实现原理混入 `AGENTS.md`。

## 不应包含

- 项目背景
- 模块职责正文
- 命令使用细节
- 长篇 workflow 说明
- 不能直接改变 agent 决策的泛化表述
